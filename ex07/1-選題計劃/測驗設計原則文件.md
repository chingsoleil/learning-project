
# IPIP 測驗設計原則

## 一、設計原則

### 1.1 科學性原則
- **信度優先**：優先選擇 `alpha` 值 ≥ 0.70 的題目，確保測驗結果的可靠性
- **效度保證**：選用經過實證研究驗證的 instruments 和 traits
- **平衡性**：每份測驗需包含正向題（key=1）與反向題（key=-1），避免反應偏差

### 1.2 多樣性原則
- **Instrument 多樣化**：5 份測驗應涵蓋不同的 instruments，避免單一來源
- **Trait 覆蓋性**：每份測驗應測量多個相關特質，提供全面的人格評估
- **題目分散性**：同一 trait 的題目應分散在測驗中，避免連續出現

### 1.3 實用性原則
- **題數固定**：每份測驗嚴格控制為 50 題，確保施測時間一致（約 15-20 分鐘）
- **主題明確**：每份測驗應有明確的主題或應用場景
- **易於解釋**：選擇易於理解和解釋的 traits，便於生成報告

### 1.4 文化適切性原則
- **語言準確**：使用已翻譯且驗證過的繁體中文題目
- **文化敏感**：避免可能因文化差異造成誤解的題目

## 四、選題原則

### 4.1 題目品質標準

1. **信度要求**
   - 優先選擇 `alpha ≥ 0.75` 的題目
   - 可接受範圍：`alpha ≥ 0.70`
   - 避免使用 `alpha < 0.70` 的題目

2. **計分方向平衡**
   - 每個 trait 應包含正向題（ScoringKey=1）與反向題（ScoringKey=-1）
   - 建議比例：正向題 60-70%，反向題 30-40%
   - **資料庫欄位名稱**：`ScoringKey`（不是 `key`）

3. **題目長度**
   - 優先選擇簡潔明瞭的題目（10-20 字）
   - 避免過長或過於複雜的題目

### 4.2 題目分配原則

每份測驗 50 題的建議分配：

| 分配方式 | 說明 | 適用場景 |
|---------|------|---------|
| **均衡分配** | 每個 trait 約 8-10 題，涵蓋 5-6 個 traits | 全面性評估 |
| **重點分配** | 核心 trait 15-20 題，輔助 traits 各 5-8 題 | 主題性評估 |
| **多維分配** | 涵蓋 8-10 個 traits，每個 5-6 題 | 廣泛性評估 |

### 4.3 題目順序原則

1. **隨機化**：同一 trait 的題目應分散，避免連續出現
2. **正向反向交替**：正向題與反向題交替出現，減少反應偏差
3. **難度平衡**：簡單題與複雜題交錯，維持受測者注意力

## 六、實施步驟

### 步驟 1：題庫篩選
1. 根據選定的 instruments 和 traits，從題庫中篩選符合條件的題目
2. 篩選條件：
   - `Alpha ≥ 0.70`（優先選擇 `Alpha ≥ 0.75`）
   - `InstrumentCategoryId` 符合選定的 instrument Id
   - `TraitCategoryId` 符合選定的 trait Id
   - `IsActive = 1`（只選擇啟用的題目）
3. **SQL 查詢範例**：
```sql
SELECT qb.Id, qb.TextZh, qb.Alpha, qb.ScoringKey
FROM QuestionBank qb
WHERE qb.InstrumentCategoryId = 17  -- Goldberg1999
  AND qb.TraitCategoryId = 64        -- Extraversion
  AND qb.Alpha >= 0.70
  AND qb.IsActive = 1
ORDER BY qb.Alpha DESC;
```

### 步驟 2：題目分配
1. 根據每份測驗的 trait 分配方案，計算各 trait 所需題數
2. 從篩選結果中選擇題目：
   - **優先策略**：按 `Alpha` 值降序排列，選擇信度最高的題目
   - **備選策略**：如果題目數量充足，可隨機選擇以增加多樣性
   - **平衡策略**：確保每個 trait 都有足夠的正向題（ScoringKey=1）和反向題（ScoringKey=-1）

### 步驟 3：題目平衡
1. 檢查每份測驗的正向題與反向題比例：
   - 統計 `ScoringKey = 1` 的題目數量（正向題）
   - 統計 `ScoringKey = -1` 的題目數量（反向題）
   - 確保比例在 60-70% : 30-40% 之間
2. 確保每個 trait 都有正向題與反向題：
   - 如果某個 trait 只有正向題或只有反向題，需補充對應方向的題目
   - 如果題庫中該 trait 沒有反向題，可考慮使用其他相關 trait 的題目

### 步驟 4：順序優化
1. 將題目隨機化，避免同一 trait 連續出現
2. 正向題與反向題交替排列

### 步驟 5：品質檢核
1. 檢查每份測驗是否為 50 題
2. 確認所有題目的 `Alpha` 值符合標準（≥ 0.70，優先 ≥ 0.75）
3. 驗證計分規則的正確性：
   - 確認 `ScoringKey` 欄位正確（1 或 -1）
   - 確認反向題的計分邏輯正確
4. 檢查題目重複：
   - 確保 5 份測驗之間沒有完全相同的題目（相同 `Id`）
   - 如果題目文字相同但 Id 不同，需確認是否為重複題目
5. 驗證資料完整性：
   - 確認所有題目都有 `TextZh`（中文題目）
   - 確認所有題目都有對應的 `InstrumentCategoryId` 和 `TraitCategoryId`

## 七、注意事項

1. **避免題目重複**：5 份測驗之間應避免使用完全相同的題目（相同 `QuestionBank.Id`）
2. **保持一致性**：使用相同的計分方式（建議 5 點 Likert 量表：1=非常不同意，2=不同意，3=普通，4=同意，5=非常同意）
3. **文化適應**：確保所有題目都有準確的繁體中文翻譯（使用 `TextZh` 欄位）
4. **資料庫欄位對應**：
   - 計分鍵：使用 `ScoringKey`（不是 `key`）
   - 信度係數：使用 `Alpha`（第一個值）和 `Alpha2`（第二個值，可選）
   - 題目文字：使用 `TextZh`（中文）和 `TextEn`（英文）
5. **特質名稱對應**：
   - 開放性：使用 "Openness To Experience" (Id=162)，不是 "Openness"
   - 自律：使用 "Self-discipline" (Id=206)，注意是小寫 d
   - 藝術興趣：使用 "Aesthetic Appreciation/Artistic Interests" (Id=6)
   - 誠實：使用 "Honesty/Integrity/Authenticity" (Id=81)
   - 謙遜：使用 "Modesty/Humility" (Id=150)
6. **常模建立**：建議收集本地常模資料，提高結果解釋的準確性
7. **倫理考量**：施測前需獲得受測者知情同意，保護個人隱私

## 附錄：資料庫查詢參考

### A. 查詢特定 Instrument 和 Trait 的題目

```sql
-- 範例：查詢 Goldberg1999 的 Extraversion 題目
SELECT 
    qb.Id,
    qb.TextZh,
    qb.TextEn,
    qb.Alpha,
    qb.ScoringKey,
    ic.NameZh AS InstrumentName,
    tc.NameZh AS TraitName
FROM QuestionBank qb
INNER JOIN InstrumentCategory ic ON qb.InstrumentCategoryId = ic.Id
INNER JOIN TraitCategory tc ON qb.TraitCategoryId = tc.Id
WHERE qb.InstrumentCategoryId = 17  -- Goldberg1999
  AND qb.TraitCategoryId = 64        -- Extraversion
  AND qb.Alpha >= 0.70
  AND qb.IsActive = 1
ORDER BY qb.Alpha DESC, qb.ScoringKey;
```

### B. 統計各 Instrument 的題目數量

```sql
SELECT 
    ic.Id,
    ic.NameEn,
    ic.NameZh,
    COUNT(qb.Id) AS QuestionCount,
    AVG(qb.Alpha) AS AvgAlpha
FROM InstrumentCategory ic
LEFT JOIN QuestionBank qb ON ic.Id = qb.InstrumentCategoryId AND qb.IsActive = 1
GROUP BY ic.Id, ic.NameEn, ic.NameZh
HAVING COUNT(qb.Id) > 0
ORDER BY QuestionCount DESC;
```

### C. 統計各 Trait 的題目數量

```sql
SELECT 
    tc.Id,
    tc.NameEn,
    tc.NameZh,
    COUNT(qb.Id) AS QuestionCount,
    SUM(CASE WHEN qb.ScoringKey = 1 THEN 1 ELSE 0 END) AS PositiveCount,
    SUM(CASE WHEN qb.ScoringKey = -1 THEN 1 ELSE 0 END) AS NegativeCount
FROM TraitCategory tc
LEFT JOIN QuestionBank qb ON tc.Id = qb.TraitCategoryId AND qb.IsActive = 1
GROUP BY tc.Id, tc.NameEn, tc.NameZh
HAVING COUNT(qb.Id) > 0
ORDER BY QuestionCount DESC;
```
